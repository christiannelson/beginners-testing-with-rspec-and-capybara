Why write specs?

  • Assert software works as specified
  • Automate monotonous and slow task of testing
  • Safty net while making changes (new featues or refactors)
  • Your responsibility as a professional developer

Why write specs first?

  • Encourages writing code that is easily tested
  • Reduces dependencies (coupling)
  • It influences the design of the code for the better: we write better code when it's test-driven

Test Driven versus Behavior Driven

BDD builds upon TDD by formalising the good habits of the best TDD practitioners.

  • Working outside-in, starting from a business or organisational goal
  • Using examples to clarify requirements
  • Developing and using a ubiquitous language

Continuous Integration (and deployment)

The practice of merging all developer workspaces into a shared mainline multiple times daily.

  1. Check out latest source
  2. Prepare the testing environment (e.g. DB)
  3. Run all of the specs
  4. Deploy to a staging environment

Can I write code without test-driving it?

Yes, when you're "spiking" on something to understand how it works or decide if it's something you want to use.

When you're done spiking, start over and test-drive the production implementation.

Common pitfalls

  1. Treating specs like 2nd class citizens
  2. Testing the same thing multiple times
  3. Focusing on coverage (the #) not the quality of coverage
  4. Slow running specs
